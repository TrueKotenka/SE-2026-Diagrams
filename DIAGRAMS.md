# Модели системы учёта дефектов и поведения микроволновой печи


## 1. Диаграмма прецедентов (Use Case)
Описывает пользователей и случаи использования разрабатываемого приложения.

```plantuml
@startuml
left to right direction
skinparam packageStyle rectangle

actor "Диспетчер" as Dispatcher
actor "Бригадир" as Foreman
actor "Ремонтник" as Worker
actor "Отдел контроля качества" as QA
actor "Начальник цеха" as Manager
actor "Бухгалтерия" as Accounting

rectangle "Система учёта дефектов" {
  usecase "Внести данные о зонах, местах и бригадах" as UC1
  usecase "Зарегистрировать дефект" as UC2
  usecase "Направить авто в свободную зону" as UC3
  usecase "Назначить ремонт рабочему" as UC4
  usecase "Зарегистрироваться доступным в начале смены" as UC5
  usecase "Зафиксировать время начала и конца ремонта" as UC6
  usecase "Получить сводный отчёт по участкам" as UC7
  usecase "Получить отчёт об общем кол-ве ремонтов" as UC8
  usecase "Получить информацию по ремонтникам бригады" as UC9
  usecase "Получить отчёт о количестве отработанных смен" as UC10
}

Dispatcher --> UC1
Dispatcher --> UC2
Dispatcher --> UC3

Foreman --> UC4
Foreman --> UC9

Worker --> UC5
Worker --> UC6

QA --> UC7
Manager --> UC8
Accounting --> UC10
@enduml

```

**Пояснение:**

* Система должна обладать возможностью вносить данные о ремонтных зонах, местах и бригадах.


* Диспетчер должен иметь возможность видеть свободные места и направлять автомобили для ремонта в свободную ремонтную зону.


* Бригадир имеет возможность назначить ремонт конкретному рабочему.


* Рабочий может зарегистрироваться в системе как доступный и зафиксировать время начала и конца ремонта.


* Различные отделы получают специфические отчёты: отдел контроля качества получает сводный отчёт по участкам , начальник цеха — отчёт об общем количестве ремонтов , бригадир — статистику по своей бригаде , а бухгалтерия — отчёт об отработанных сменах.

## 2. Диаграмма активностей (Activity Diagram)

Описывает основной бизнес-процесс, поддерживаемый приложением — регистрацию и ремонт дефекта.

```plantuml
@startuml
|Отдел контроля качества|
start
:Выявление дефекта сборки;
:Снятие автомобиля с конвейера;

|Диспетчер|
:Просмотр свободных мест в ремонтных зонах;
:Направление автомобиля в свободную ремонтную зону;

|Ремонтник (Диагност)|
:Поступление автомобиля в ремонтную зону;
:Выполнение первоначальной диагностики;
:Регистрация дефекта в системе;

|Бригадир|
:Назначение ремонта доступному рабочему;

|Ремонтник (Исполнитель)|
:Фиксация времени начала ремонта;
:Устранение дефекта;
:Фиксация времени конца ремонта;

|Отдел контроля качества|
:Возврат автомобиля в зону тестирования или на конвейер;
stop
@enduml

```

**Пояснение:**

* В случае обнаружения дефекта автомобиль снимается с конвейера и отправляется в ремонтную зону.


* Диспетчер направляет автомобиль для ремонта в свободную ремонтную зону.


* Дефект регистрируется после поступления автомобиля в ремонтную зону и выполнения первоначальной диагностики.


* После устранения дефекта автомобиль возвращается на конвейер либо в зону тестирования.

## 3. BPMN-диаграмма (Бизнес-процесс завода)

BPMN-диаграмма для всего бизнес-процесса завода, включая внешних его участников. В PlantUML для этого используются дорожки (swimlanes) диаграммы активности.

```plantuml
@startuml
|Дилер|
start
:Отправка набора заказов на автомобили;

|Отдел планирования|
:Составление производственного плана;

|Производство|
:Сборка кузова;
:Покраска (несколько дней);
:Сборка остальных комплектующих на конвейере;

|Отдел контроля качества|
:Стендовые и ходовые испытания;
if (Выявлены дефекты?) then (Да)
  |Ремонтная зона|
  :Снятие с конвейера и отправка в ремонт;
  :Устранение дефекта;
  |Отдел контроля качества|
else (Нет)
endif

|Дилер|
:Отгрузка автомобиля заказчику;
stop
@enduml

```

**Пояснение:**

* Завод получает набор заказов от дилеров.


* Отдел планирования составляет производственный план.


* Процесс производства включает сборку кузова, покраску и сборку остальных комплектующих.


* В отделе контроля качества автомобиль проходит серию стендовых и ходовых испытаний.


* В случае успешного прохождения проверок автомобиль отгружается заказчику.

## 4. Диаграмма классов (Class Diagram)

Диаграмма классов, моделирующая данные, хранимые системой.

```plantuml
@startuml
class Автомобиль {
  +vin: String
  +комплектация: String
  +цвет: String
  +датаПоступления: Date
}

class Дефект {
  +местоНаСхеме: String
  +возможнаяПричина: String
}

class Ремонт {
  +времяНачала: DateTime
  +времяКонца: DateTime
}

class РемонтнаяЗона {
  +названиеУчастка: String
}

class РемонтноеМесто {
  +номерМеста: Integer
  +статусСвободно: Boolean
}

class Бригада {
  +номерБригады: Integer
  +смена: String
}

class Сотрудник {
  +фио: String
  +роль: String
  +статусДоступен: Boolean
}

Автомобиль "1" -- "*" Дефект : имеет >
Дефект "1" -- "1" Ремонт : требует >
РемонтнаяЗона "1" *-- "1..4" РемонтноеМесто : содержит >
РемонтнаяЗона "*" -- "1" Бригада : обслуживается >
Бригада "1" *-- "*" Сотрудник : состоит из >
Сотрудник "1" -- "*" Ремонт : выполняет >
Ремонт "*" -- "1" РемонтноеМесто : проводится на >
@enduml

```

**Пояснение:**

* Система хранит данные о ремонтных зонах, ремонтных местах, ремонтных бригадах и их бригадирах.


* Каждый участок конвейера имеет несколько (от одной до четырёх) ремонтных зон, в каждой зоне имеется несколько ремонтных мест.


* Каждая ремонтная зона обслуживается ремонтной бригадой, имеющей бригадира и нескольких ремонтников.


* При регистрации дефекта указывается место дефекта на схеме, возможная причина, номер автомобиля, дата поступления и выполнявший диагностику ремонтник.
